## 第一轮评审

### Spec 12（Auto-Use A）

#### P0

- [x] **S12-P0-1 `InstallOp` union type 缺 `kind:"cli"` variant**

  **位置**：Spec 12 Task 5 接口定义与 claude-code adapter 描述之间存在矛盾。

  Task 5 在 `ClientAdapter` 设计里写道：
  > "claude-code 优先 `claude mcp add`（探测到 `claude` CLI 时，plan 产出一个 `kind:"cli"` op；否则降级 `json-mcp` 写 `~/.claude.json`）"

  但 Task 5 紧接着给出的 `InstallOp` 类型定义只有四种 kind：
  ```ts
  type InstallOp =
    | { path: string; kind: "json-mcp"|"toml-mcp"; action: "upsert"|"remove"; entry?: McpEntry }
    | { path: string; kind: "marked-block"|"managed-file"; action: "upsert"|"remove"; content?: string };
  ```
  `kind:"cli"` 不在其中，但 orchestrator（Task 6）需要按 kind 分派执行 —— 未声明的 kind 会导致 TypeScript 类型错误，orchestrator 对 `cli` op 也没有执行分支。

  **需修复**：`InstallOp` 补充第三个 variant：
  ```ts
  | { kind: "cli"; action: "upsert"|"remove"; args: string[] }
  ```
  并在 orchestrator 里处理 `kind:"cli"` → 子进程调用 `claude mcp add/remove` 的执行分支。

### Spec 13（Auto-Use B）

#### P0

- [x] **S13-P0-1 REST 搜索路径的 `mode=fts` 参数不存在**

  **位置**：Spec 13 Task 2 `recall-client.ts` 设计、`实测 API / 现状` 小节。

  Spec 13 写：
  > "优先 `GET serve/{search-route}?mode=fts&q=...`"

  代码核查（`src/server/api.ts`）：`GET /search` 接受 `q`、`type`、`exclude_types`、`from`、`to`、`platform`，**无 `mode` 参数**。传入 `mode=fts` 会被静默忽略，实际执行的仍是默认 search 逻辑（不确定是否是 FTS-only 还是 hybrid）。

  这直接影响 Spec 13 核心设计目标：UserPromptSubmit hook 要求 **零成本 FTS**（不触发 embedding 调用），若 REST `/search` 默认走 hybrid 检索，每条 prompt 都会触发向量化，违背「cheap-first」原则。

  **需在修复前先确认**：
  1. `GET /search` 当前是 FTS-only 还是 hybrid？（检查 `SearchStore.search()` 实现）
  2. 若已是 FTS-only：去掉 `mode=fts`，改为 `GET /search?q=...`，并在 spec 注明「/search 是 FTS-only，无需 mode 参数」。
  3. 若是 hybrid：需扩展 `GET /search` 支持 `?mode=fts`（仅跑 FTS 腿，跳过 embedding），或 recall-client 改走直连 store 的 FTS path（`store.ftsSearch(q)`）。

#### P1

- [x] **S13-P1-1 hook 输出格式 `hookSpecificOutput` 无仓库内依据**

  **位置**：Spec 13 Task 1、Task 2 的 handler 设计。

  Spec 写：
  > "规范用 `{ \"hookSpecificOutput\": { \"hookEventName\":\"...\", \"additionalContext\":\"...\" } }`"

  仓库内当前**无 Claude Code hook 实现**（`.claude/settings.json` 不存在，代码中无任何 hook handler），无法从已有实现交叉验证此格式。若格式错误（如 CC 当前版本用 `{ "context": "..." }` 或直接 stdout 纯文本），两个读侧 hook（SessionStart + UserPromptSubmit）产出的 JSON 将被 Claude Code 忽略，功能全部失效。

  **需修复**：
  1. Task 1 之前加一步「格式调研」：从 Claude Code 当前文档（`claude hooks --help` 或 CC release notes）确认 hook stdout 协议，锁定输出格式版本。
  2. 将锁定格式写入 Spec，让 `handlers.ts` 实现有确定依据。

- [x] **S13-P1-2 注入预算 `≤800 token` 未指定计量方式**

  **位置**：Spec 13「注入预算（本 spec 锁定）」小节、Task 2 Step 1 测试。

  > "注入 ≤ ~800 token"

  Bun/Node.js 运行时无原生 tokenizer（Tiktoken 依赖 WASM，cl100k 词表），实现时无明确方法计量 "800 token"。不同语言（中文 1 字 ≈ 0.5~1.5 token）误差大。

  **需修复**：改为字符预算，如 `≤ 3000 chars`（约等于 ~750 token @ 4 chars/token 粗估），并明确「按 `text.length` 截断，不引 tokenizer 依赖」。或若接受外部依赖，显式声明引入哪个轻量 tokenizer。

### Spec 14（Auto-Use C）

#### P1

- [x] **S14-P1-1 `MEMOARK_FOR_AGENTS.md` 引用了不存在的命令 `memoark start`**

  **位置**：Spec 14 Task 3 Step 2-3 描述（`MEMOARK_FOR_AGENTS.md` 内容设计）。

  Spec 写：
  > "②无配置则 `memoark start` 引导"

  代码核查（`src/cli.ts`）：**不存在 `memoark start` 命令**。现有命令为 `memoark serve`（启动服务）和 `memoark init`（初始化数据库）；Web Config UI 对应 `memoark init --web`（Spec 5，待实现）。

  Agent 按此剧本跑 `memoark start` 会报「unknown command」，自安装流程中断。

  **需修复**：改为 `memoark init`（若 DB 未初始化）或 `memoark init --web`（若 Spec 5 已实现时引导 UI 配置）；在 Task 3 头部注明该命令依赖 Spec 5 实现状态。

#### P2

- [x] **S14-P2-1 Hermes 配置目录 `~/.hermes` 与仓库内探测路径不一致，需核实**

  **位置**：Spec 14 Task 2 hermes adapter、`实测 API / 现状` 小节。

  `src/setup/detect-sources.ts` 探测 Hermes session 日志的路径为 `~/.openclaw/agents`（Hermes 历史上基于 OpenClaw）。Spec 14 的 hermes adapter 则通过 `~/.hermes` 是否存在来 `detect()`，并向 `~/.hermes/config.yaml` 写 MCP 配置、向 `~/.hermes/skills/` 铺 skill。

  两者前缀不同（`~/.openclaw` vs `~/.hermes`）。可能场景：
  - Hermes 在新版本中将主目录迁至 `~/.hermes`（detect-sources 未更新）；或
  - Hermes config 在 `~/.openclaw/config.yaml`，`~/.hermes` 根本不存在。

  若 `~/.hermes` 不存在，`detect()` 永远返回 false，`memoark install --agent hermes` 静默无操作。

  Spec 已说「实现时核对 Hermes 当前 config schema」，建议同时核实：目录是 `~/.hermes` 还是 `~/.openclaw`，并补一句「若实测 `~/.openclaw`，detect 改用该路径，config 路径同步更新」。
