- [x] **S10-P1-4 `search.pool_by_page` 配置项未说明 schema 扩展**  
  §三提到增加 `search.pool_by_page` 配置，但 `memoark.yaml` 的 Zod schema（`src/core/config.ts`）需同步更新。Spec 10 未提及此步骤。

- [x] **S10-P1-5 poolByPage 从 `scope.ts` 移入 `search.ts` 存在双重池化风险**  
  Spec 7 §七已在 `src/synth/scope.ts` 实现 `poolByPage`（调 `hybridSearch` 后再 reduce）。Spec 10 §三说改 `hybridSearch` 默认 `poolByPage:true`，意味着池化进入 `search.ts` 内部。需明确：(A) 把池化从 `scope.ts` 移到 `hybridSearch` 内，`scope.ts` 不再二次 reduce；还是 (B) 两层并存（可能双重池化）。必须选一，否则行为不确定。

- [x] **S10-P1-6 `links.provenance` 列类型未核实**  
  "写 `{ auto: 'wikilink' }` 到 `provenance`"——`links.provenance` 若是 TEXT 列则不能直接存 JSON 对象。需核 `schema.sql` 中 `links` 表 `provenance` 列类型（TEXT vs JSONB），再确定存储格式。

### Spec 8

- [ ] **R2-S8-P1-1 `pinned context block` 修改了 Spec 7 的 `AssembledContext`，但未声明扩展点**  
  §6.1 修复 S8-P1-2 的方案是"scope.ts entity-scope 检索统一加一步——若 `frontmatter.profile` 存在，把它渲染成 pinned 上下文块置于候选 ref 之前"。但这引入了一个 Spec 7 从未定义的概念：Spec 7 的 `AssembledContext`（`context.ts`）只有 `candidates: AssembledCandidate[]`，**没有 `pinnedContext` 字段**。
  
  **需要明确**：  
  (A) `AssembledContext` 加 `pinnedContext?: string` 字段（需同步更新 Spec 7 §3.2 data contracts）；  
  (B) 实现这步的是 **Spec 7 的 scope.ts**（entity-scope 通用逻辑）还是 **Spec 8 修改 scope.ts**（profile 专有逻辑）；  
  (C) `pinned block` 是否有 `ref` 编号（如果没有，LLM 无法引用它，画像就只是上下文而非可引用证据）。  
  
  不指定清楚，实现者要么在 Spec 7 里硬编码"如果有 profile 就注入"（耦合），要么在 Spec 8 里改 scope.ts（需要明确列为 Spec 8 的 scope 文件修改）。

### Spec 9

- [ ] **R2-S9-P1-1 `dailyReportIntent.systemPrompt` 仍是代码注释占位符**  
  §5.1 修复了 `parseSections`（✅），但意图对象里 `systemPrompt` 字段至今是：  
  ```typescript
  systemPrompt: /* §5.3：要求 LLM 用 "## 段标题" 输出 7 段 */,
  ```  
  这是 JavaScript 注释，**不是字符串**——TypeScript 编译器会报错，测试无法运行，实现者也不知道实际 prompt 内容。`parseSections` 依赖 LLM 输出精确的 7 个 H2 标题（中文名称必须固定），否则切分会出错。
  
  **需要补充**：具体的 systemPrompt 字符串，至少包含：要求使用 `## <段名>` 格式、列出 7 个固定中文段名（如"今日概览""今日完成"……）、要求用 `[n]` 标注来源。

### Spec 11

- [ ] **R2-S11-P1-1 `precedes` 预排序在 `scope.ts` 里执行，但 scope.ts 是 Spec 7 的通用文件，与 Spec 11 形成单向耦合**  
  §6.1 修复 S11-P0-2 的方案是"scope.ts 对 troubleshoot 检索结果按 `precedes` 链预排序（用 §三 的 `getOrderedSequence`）"。问题：`scope.ts` 是 Spec 7 的通用检索层，`getOrderedSequence` 是 Spec 11 里的 graph 查询——**Spec 7 的文件不应该 import Spec 11 的函数**。  
  
  **二选一（需锁定）**：  
  (A) Spec 7 的 `IntentTemplate` 增加可选钩子 `sortCandidates?(candidates: AssembledCandidate[], stores: Stores): Promise<AssembledCandidate[]>`（推荐）——`troubleshootIntent` 在 Spec 11 里实现此钩子，调 `getOrderedSequence`，scope.ts 通用调用钩子，零耦合；  
  (B) scope.ts 仅对 `types.includes("playbook")` 的请求做 `precedes` 排序，不 import 具体意图（type-driven，比 intent-driven 好，但仍是 scope.ts 特殊逻辑）。  
  
  如选 (A)，Spec 7 的 `IntentTemplate` 接口和 `engine.ts` 的 synthesize 流程需同步更新。
